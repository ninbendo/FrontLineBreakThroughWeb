# AI Eval Runbook

## 目的
AI 実装支援（Copilot / Cursor / Claude Code / Codex など）を、同じ観点・同じ進め方で比較できる形で記録する。

## このRunbookの位置づけ
このファイルは **共通ルールと記録テンプレート** を置く。
各AIごとの実施結果は `docs/ai-evals/` 配下の個別ファイルに記録する。

例:
- `docs/ai-evals/copilot.md`
- `docs/ai-evals/codex.md`
- `docs/ai-evals/claude-code.md`

## 評価の基本方針
- 同じ課題を使って比較する
- できるだけ同じ粒度の指示で試す
- 実装差分は必ず人間がレビューする
- AIごとの得意・不得意を、印象ではなく記録で残す
- 評価ブランチ上で試し、必要なものだけ main に反映する

## 評価課題
- テーマ: ③ゲート値更新
- 狙い: 純ロジック切り出しと Edit Mode テスト化の適性を見る
- 実装対象候補:
  - GateController
  - 新規 Rules / Calculator クラス
  - Edit Mode テスト

## 参照資料
- `docs/specs/1.ゲーム要件.txt`
- `docs/specs/2.ゲーム設計書.txt`
- `docs/specs/3.技術設計メモ（クラス責務・参照関係）.txt`
- `docs/specs/4.実装タスク票.txt`
- `docs/specs/5.受け入れテスト表.txt`
- `docs/specs/6.自動テスト方針メモ（Unity Test Framework導入用）.txt`

## 比較観点
- 読み取り精度
- 指示遵守
- 小変更の安全性
- Unity文脈との整合
- 不要変更の少なさ
- テストの妥当性
- 修正往復回数
- 導入しやすさ
- コスト感

## 推奨の進め方
1. 読み取りだけ
2. 設計提案
3. 変更前の最終しぼり込み
4. 最小差分実装
5. 差分レビュー
6. 修正指示
7. 再レビュー

## 記録ルール
- 実施AIごとに `docs/ai-evals/` 配下へ1ファイル作成する
- 結果は `Pass / Partial / Fail / N/A` のいずれかで記録する
- 良かった点と問題点は、できるだけ具体例ベースで書く
- 不採用にした差分も、理由を残す
- 比較しやすいよう、記録項目の見出しはなるべく揃える
- 比較観点は原則として共通の9項目を使い、省略する場合は理由を残す
