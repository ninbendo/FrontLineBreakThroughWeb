# ⑥自動テスト方針メモ v1.1（Unity Test Framework導入用）

## FrontLineBreakThroughWeb（MVP）

## 1. 目的

* MVP開発で壊れやすい仕様を、自動テストで早期検知できるようにする
* ⑤受け入れテスト表（手動テスト）を上位基準として、再発防止しやすい形で自動テストへ落とし込む
* 将来の本仕様（MVP以降）へ拡張しやすいテスト基盤を作る

---

## 2. 位置づけ（重要）

### 2-1. 本資料の役割
* 本資料は「どの仕様を、どのテスト種別で、自動化するか」を決める方針書である
* 実際のテストコード（NUnit / Unity Test Framework）は別タスクで作成する

### 2-2. ⑤受け入れテスト表との関係
* ⑤受け入れテスト表：人間が操作して確認する最終確認（受け入れ）
* ⑥自動テスト方針メモ：⑤のうち、自動化効果が高い項目を抽出してテスト種別を割り当てる
* 原則として、受け入れテストで見つかった不具合は再発防止のため自動テスト化を検討する

---

## 3. Unity Test Framework 前提（MVP方針）

### 3-1. 使用する仕組み
* Unity Test Framework（UTF）
* NUnitベース
* Test Runner（Unity Editor内）を利用

### 3-2. テスト種別の使い分け方針
#### A. Edit Mode テスト（高速・純ロジック向け）
* 目的：計算・分岐・クランプ・状態遷移などの純ロジック確認
* 使う場面：
  * 人数増減の上限/下限（描画上限89人 / 0未満防止）
  * 内部カウント上限なし（描画上限超過時も増え続ける）
  * 武器Lv上限
  * ゲート値更新（1発=1変化）
  * ゲート色判定（負数=赤、0以上=青）
  * 状態遷移関数（Playing/Clear/GameOver）

#### B. Play Mode テスト（ランタイム挙動向け）
* 目的：シーン上の挙動、フレーム経過、Trigger/Collision、コンポーネント連携確認
* 使う場面：
  * 前進感（障害物が奥から手前に流れる） / 自動射撃
  * ゲート接触時の人数反映（1回だけ）
  * ゲート色動的変化（数値変化のたびにリアルタイム更新）
  * タル接触ペナルティ
  * 護衛兵取得で人数増加
  * 敵HP減少・撃破（骨格）
  * ゴール / 全滅の状態遷移

#### C. 手動受け入れテスト（⑤のまま残す）
* 目的：体感、見た目、実機、公開環境確認
* 使う場面：
  * 3D見下ろし斜め視点・3レーン構造の見た目確認
  * スマホ実機操作
  * レイアウト見切れ
  * Xserver公開確認
  * 音ON/OFFの体感

---

## 4. 自動テスト導入の基本方針（重要）

### 4-1. いきなり全部を自動化しない
* まずは「壊れると痛い」「仕様差分が出やすい」ロジックから自動化する
* 見た目の良し悪し・体感は ⑤手動受け入れテストに残す

### 4-2. 純ロジックを分離してEdit Modeでテストしやすくする
* MonoBehaviour に計算ロジックを詰め込みすぎない
* 可能ならルール計算をプレーンC#へ切り出す
  * 例：人数クランプ、武器Lv更新、ゲート値更新、状態遷移判定

### 4-3. Play Mode テストは「仕様の芯」だけに絞る
* Trigger/Collision/フレーム進行を伴う重要仕様のみ
* UI見た目の細かい検証は受け入れテストで行う

### 4-4. バグ修正時のルール
* 受け入れテストでFail/Partialになった不具合は、再発防止のため以下を検討する
  1. Edit Modeで再現できるか
  2. 無理ならPlay Modeで再現する
  3. それでも難しいなら手動受け入れケースを強化する

---

## 5. テスト対象の優先順位（MVP）

## 優先度A（最初に自動化する）
1. 人数増減クランプ（描画上限89人 / 0未満防止）
2. 内部カウント上限なし（描画上限超過時も増え続ける）
3. 武器Lv上限（Lv3固定）
4. ゲート値更新（1発=1変化、負→0→正）
5. ゲート色判定（負数=赤、0以上=青）
6. ゲート接触時の人数反映が1回だけ
7. 状態遷移（Clear / GameOver）

## 優先度B（次に自動化）
8. 前進感（障害物が奥から手前に流れる）
9. 自動射撃（発射間隔の概略）
10. ゲート色動的変化（ランタイムでの色切替）
11. タル接触で1体減少（OnTriggerEnter基準）
12. タル破壊→固定ドロップ（護衛兵 / 機関銃）
13. 護衛兵取得で人数増加
14. 敵HP減少・撃破（骨格）
15. リトライで初期状態へ戻る（最小確認）

## 優先度C（手動中心で運用）
16. 3D見下ろし斜め視点の見た目
17. 3レーン構造の見た目
18. レイアウト見切れ
19. スマホ実機操作感
20. 音ON/OFF体感
21. Xserver公開確認

## 7. MVPで最初に作る自動テスト（推奨セット）

## 7-1. Edit Mode テスト（最初の4本）
### E-001 人数クランプ（下限）
* 入力：現在人数=1、減少量=5
* 期待：結果人数が0未満にならない（0で止まる or GameOver判定値へ正規化）
* 関連：T-035, T-044

### E-002 描画上限と内部カウントの分離
* 入力A：現在人数=88、増加量=5
* 期待A：描画人数は89で頭打ち、内部カウントは93になる
* 入力B：現在人数=100（内部）、さらに増加量=50
* 期待B：内部カウントは150、描画人数は89のまま
* 関連：T-044, T-044.5

### E-003 武器Lv更新（Lv3上限）
* 入力：Lv1→取得→Lv2→取得→Lv3→取得
* 期待：Lv3で頭打ち
* 関連：T-061, T-062, T-063

### E-004 ゲート色判定（負数=赤、0以上=青）
* 入力A：ゲート値=-1
* 期待A：色=赤
* 入力B：ゲート値=0
* 期待B：色=青
* 入力C：ゲート値=+1
* 期待C：色=青
* 関連：T-037

## 7-2. Play Mode テスト（最初の2〜3本）
### P-001 ゲート接触時の人数反映が1回だけ
* ゲート接触 → 再接触 / 継続接触
* 期待：同一ゲートの反映は1回のみ
* 関連：T-033, T-034, T-036

### P-002 タル接触で1体減少（OnTriggerEnter基準）
* 1回接触
* 期待：兵隊数が1だけ減る
* 関連：T-054

### P-003 ゴール到達でClear / 全滅でGameOver
* 期待：GameManager状態遷移が正しい
* 関連：T-080, T-081

## 8. テストコード設計のルール（運用ルール）

### 8-1. 命名規則（例）
* Edit Mode：`XxxRulesTests`, `XxxCalculatorTests`
* Play Mode：`XxxPlayModeTests`, `GameFlowPlayModeTests`

### 8-2. 1テスト1責務
* 1つのテストで複数仕様を同時に検証しすぎない
* Fail時に原因が切り分けられる粒度にする

### 8-3. 期待値は仕様文言に寄せる
* 「何となく動いた」ではなく、仕様に対応した期待値にする
* 可能ならテストコメントに ⑤テストID を書く
  * 例：`// Covers T-063`

### 8-4. Play Mode テストの注意
* フレーム待ちが必要な箇所だけ `UnityTest` を使う
* 待ち時間を長くしすぎない（テスト全体が遅くなる）
* Time依存を減らせるならロジック分離してEdit Modeへ寄せる

---

## 9. 導入ステップ（実務向け）

### Step 1：UTFを有効化し、テスト実行確認
* Test Runnerで Edit Mode / Play Mode の空テストを1本ずつ実行
* 「まず動く」状態を作る

### Step 2：asmdef整理（必要に応じて）
* テスト対象コードを参照できるよう asmdef 構成を整理
* テストしたいコードを `Assembly-CSharp` 依存のままにしない設計を検討
  * （将来的に自動テストしやすくするため）

### Step 3：Edit Modeテストから着手（優先度A）
* 人数クランプ / 武器Lv上限 / ゲート値更新

### Step 4：Play Modeテストを追加
* ゲート1回反映 / タル接触 / Clear/GameOver

### Step 5：⑤受け入れテスト表と運用連携
* 手動でFailになったケースを、再発防止として自動テスト候補へ追加

---

## 10. 受け入れテスト（⑤）との運用ループ

1. ④実装タスクを進める
2. ⑤受け入れテストを実施（手動）
3. Fail / Partial を記録
4. 原因を切り分け（仕様差分 / バグ / 未実装）
5. 再発防止価値が高いものを ⑥ に追記（自動テスト候補化）
6. UTFで自動テストを追加
7. 次回以降の確認コストを下げる

---

## 11. MVP時点での「自動化しすぎない」判断基準

以下は、MVP段階では無理に自動化しなくてよい：

* 見た目の美しさ（配置が“気持ちいい”か）
* 音の体感品質
* 実機端末ごとの差異（ブラウザ挙動）
* Xserverの公開手順そのもの

理由：
* 自動化コストが高い
* 仕様変更がまだ入りやすい
* 手動受け入れの方が早く確実な場合が多い

---

## 12. 次アクション（推奨）

### 12-1. まず作るファイル（最小）
* Edit Mode テスト：人数クランプ / 描画上限・内部カウント分離 / 武器Lv上限 / ゲート値更新 / ゲート色判定
* Play Mode テスト：ゲート1回反映 / Clear/GameOver

### 12-2. 先に決めること（短くてよい）
* 人数クランプの正規仕様（0で止める / マイナスを許容してからGameOver判定 など）
* 描画上限（renderCap）と内部カウント（internalCount）の管理責務をどこに置くか
* ゲート値更新ロジック・色判定ロジックの責務をどこに置くか（GateController内 or ルールクラス切り出し）
* GameManager状態遷移APIの最終形（最低限のpublic API）

---

## 付録A：テスト種別の判定ルール（迷った時）

### Edit Mode に寄せる条件
* シーン不要
* フレーム経過不要
* Trigger/Collision不要
* 入出力が数値・状態で表現できる

### Play Mode にする条件
* Scene / GameObject / Component 実体が必要
* Trigger/Collisionを使う
* フレーム経過で検証する
* 実際のランタイム挙動を確認したい

### 手動に残す条件
* 実機体感
* 見た目評価
* 公開環境確認
* 操作の分かりやすさ評価
